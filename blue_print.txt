Kurukuru Golf Cart Remote Monitoring System
1. Final System Goal

Build a stable desktop monitoring system for a city office to monitor 2 moving golf carts in real time.

Each cart has:

1 front camera
1 internal camera
1 GPS/GNSS source
1 LTE/SIM router

Monitoring room has:

Ubuntu receiving PC for video wall
Windows PC or same Ubuntu PC for map
External display support

Main target:

4-camera live view
low latency
low bitrate
GPS map tracking
simple operator UI
stable reconnect
auto-start
easy maintenance
2. Final Technology Stack
Desktop App: Electron + React + TypeScript
Video Player: mpv
Camera Protocol: RTSP low-latency H.264
Map: Mapbox GL JS
GPS: SE220 GNSS → local receiver
Network: WireGuard VPN
Database: SQLite
ORM: Prisma
Optional Stream Relay: MediaMTX
OS Target: Ubuntu receiving PC
3. Recommended Video Setting

Client does not need Full HD in 4-layout.

Use this:

Resolution: 1280×720
FPS: 10–15 fps
Codec: H.264
Bitrate: 1.2–2.0 Mbps per camera
Keyframe interval: 1 sec
Audio: Off

Maximum traffic:

1 camera = max 2 Mbps
1 vehicle = 2 cameras = max 4 Mbps upload
2 vehicles = 4 cameras = max 8 Mbps total

Recommended normal setting:

720p / 15fps / 1.5 Mbps

This is realistic for LTE.

6. SQLite Usage

SQLite is OK.

But use it correctly:

Realtime GPS display = memory + WebSocket
History/settings/logs = SQLite

Do not make the map read SQLite every second.

Correct flow:

SE220 GPS data received
→ parse GPS
→ update latest vehicle location in memory
→ send to map by WebSocket
→ save GPS log to SQLite

9. mpv Low-Latency Settings

Use mpv like this:

mpv rtsp://user:pass@camera-ip/stream \
  --profile=low-latency \
  --untimed \
  --no-cache \
  --cache=no \
  --demuxer-lavf-o=rtsp_transport=tcp

Alternative for lower latency:

--demuxer-lavf-o=rtsp_transport=udp

Use TCP first for stability.
Test UDP if delay is too high.

10. Network Design

Recommended:

WireGuard VPN

Why:

- Secure
- Stable
- No public camera ports
- Works well with Linux
- Easy to maintain

Do not expose RTSP directly to internet.

Vehicle router should connect to VPN.

Monitoring PC sees cameras like private IPs:

Vehicle 1 Front:    10.10.1.11
Vehicle 1 Internal: 10.10.1.12
Vehicle 2 Front:    10.10.2.11
Vehicle 2 Internal: 10.10.2.12
11. GPS Design

SE220 GNSS sends GPS data.

Need to confirm actual output method:

NMEA0183 over TCP
NMEA0183 over UDP
Router API
Serial-to-network forwarding

Expected NMEA example:

$GPRMC,...
$GPGGA,...

GPS receiver service:

1. listen for GPS data
2. parse NMEA
3. identify vehicle
4. update memory
5. broadcast to Mapbox window
6. save to SQLite
12. Reconnect Logic

Very important.

Camera behavior:

If stream fails:
  show "Reconnecting..."
  retry after 2 sec
  retry after 5 sec
  retry after 10 sec
  keep retrying

GPS behavior:

No GPS for 5 sec:
  show delayed warning

No GPS for 15 sec:
  show offline

GPS returns:
  automatically recover

App behavior:

If app crashes:
  systemd restarts app

If PC reboots:
  app starts automatically

17. Proposal Wording

Use this wording:

The system is designed as a low-latency remote monitoring system for municipal golf carts. It provides live camera monitoring and real-time GPS map display using the existing receiving PC and displays.

The camera streams are optimized for stable LTE operation using low-bitrate H.264 RTSP video. The system targets approximately 1 second video delay under normal network conditions, with automatic reconnection when mobile signal quality changes.

Do not promise:

Always under 1 second

Say:

Target approximately 1 second under normal LTE conditions.
